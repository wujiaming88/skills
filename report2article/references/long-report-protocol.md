# 超长报告强制处理协议

**强制要求**：报告再长，信息也不能丢。这是红线的延伸。

## 为什么需要它（两个失败模式）

| 失败模式 | 现象 | 后果 |
|---|---|---|
| **输入截断** | 报告太长，一次读不全，模型只看到前半段就开始编排 | 后半段对象/数据静默丢失 |
| **输出截断** | 编排后长文超单次输出上限（stopReason=length） | 文章写一半就断，尾部对象消失 |

光靠"注意别丢"挡不住。必须用**分块 + 销账**的机械流程。

## 何时触发本协议

满足任一条即按本协议处理（宁可多走流程，不可冒丢失风险）：
- 报告对象数 ≥ 8 个，或
- 报告正文预估 ≥ 6000 字 / ≥ 8000 token，或
- 一次读取被工具截断（read 提示 more lines / API 上限）。

## 五步强制流程

### Step 1：先建全量对象清单（处理前的"分母"）
- 分段读全报告（read 用 offset/limit 翻完，**绝不只读前半段**）。
- 列出全部对象/数据点编号清单：`#1 …  #2 …  … #N`。
- 这份清单是后面每一块的"销账依据"。记下总数 N。

### Step 2：分块切分
- 按报告的逻辑章节/分组把对象切成若干块（每块 2–4 个对象为宜）。
- 给每块标注它覆盖清单里的哪些编号（如"块二 = #4~#7"）。

### Step 3：分块编排 + 立即落盘（抗输出截断）
- 一次只编排一块，**编排完立即 write/append 写到输出文件**。
- 长文用**多次 append**累加，绝不囤在上下文里等最后一次性输出整篇。
- 每块落盘后确认文件确实增长（wc -l / tail 看尾部）。

### Step 4：逐块销账
- 每落盘一块，更新覆盖计数："已覆盖 X/N 对象（#编号清单）"。
- 处理到下一块前，确认上一块的所有编号都已写入文件。
- 缺哪个编号 → 回去补哪个，补完才推进。

### Step 5：末尾一致性复核
- 全文落盘后，对照 Step 1 清单跑完整的信息零丢失 5 维校验。
- **必须 X == N**（清单总数 = 文章覆盖数）。
- 用 grep 机械核对：对象名/关键数据/链接在输出文件里逐个 `grep` 命中。
- 报出：`信息校验：N 对象 / M 数据点 / K 结论 / L 链接，全部对应 ✅`。

## 机械核对小抄

```bash
# 对象数（子标题）
grep -c '^### ' 输出文件
# 四段式完整性（每个对象应各 1 段）
for k in 本周动态 关键数据 原文链接 影响判断; do echo "$k: $(grep -c "$k" 输出文件)"; done
# 链接总数（与原报告比对，不得减少）
grep -o 'https://[^ ；）]*' 输出文件 | wc -l
```

## 红线提醒

- 分块是**抗截断手段**，不是"写短/精简"的借口。每个对象该多深写多深，原文四段式逐字保留。
- 分块只改"分几次写"，不改"写什么"——编排层动作不变，内容层一字不动。
